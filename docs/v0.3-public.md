# CannyForge v0.3 Public Summary

This document is the public-facing summary for the v0.3 direction on the benchmark branch.
It is intentionally lighter than the internal execution plan and focuses on adoption, use cases,
and developer value.

## What v0.3 Adds

CannyForge v0.3 sharpens the product around one core promise: tool-using agents should improve
from real mistakes, with measurable results and portable learning.

The push is centered on three areas:

1. Better agent improvement loops
   CannyForge records tool-use failures, learns actionable corrections, and injects those
   corrections before future model calls.

2. Better benchmark coverage
   Evaluation expands beyond first-tool selection to include argument quality, multi-step tool
   order, and recovery behavior.

3. Better portability
   Learned corrections can be exported and reused across workflows, assistants, and environments.

## Why This Matters

Many agent stacks still rely on static prompts and one-shot tool routing. That works for clean
demo cases, but breaks down when real tasks require:

- multi-step execution
- argument formatting discipline
- context-sensitive follow-up actions
- repeated tool-use recovery over time

v0.3 makes CannyForge more useful for teams that want a lightweight reliability layer instead of
model retraining or increasingly large system prompts.

## Coverage Areas

The benchmark branch is aimed at developers building agents in several common patterns:

| Area | Public story |
|------|--------------|
| LangGraph agents | Middleware-based correction injection and outcome tracking |
| LangChain-style tool agents | Thin adapter path for tool execution and improvement loops |
| CrewAI-style orchestration | Adapter surface for multi-agent task runners |
| MCP-connected assistants | MCP server path for assistant workflows and tools |
| Personal assistants | Reusable skills such as calendar, email, summarization, and web search |

## Public Positioning

The benchmark branch should present CannyForge as:

"A reliability memory layer for tool-using agents that improves routing, arguments, and
multi-step behavior over time."

That framing is broader and more durable than a narrow benchmark claim.

## What We Should Show Publicly

For public repo readers, the most useful signals are:

- clear quickstart integration for LangGraph
- evidence that learning improves real tool-use behavior
- framework coverage across modern agent stacks
- examples that look like real assistant workflows
- portable skill bundles that can move between environments

## What We Should Avoid Publishing in Detail

This branch should avoid over-sharing internal release mechanics such as:

- exact internal success thresholds
- detailed benchmark gaming strategies
- full adversarial prompt design
- private lifecycle heuristics for correction pruning

Those are useful internally, but they are not necessary for public adoption.

## Suggested Public Branch Message

Use a short message consistently across README, release notes, and benchmark materials:

"CannyForge helps tool-using agents improve from real mistakes. v0.3 expands evaluation beyond
tool selection, improves correction reliability, and makes learned skills portable across
frameworks and assistant workflows."
